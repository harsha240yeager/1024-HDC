1. What is Hyperdimensional Computing?

Hyperdimensional Computing (HDC), also called Vector Symbolic Architecture, is a brain-inspired computing model that represents every piece of information — a sensor reading, a feature, a class label — as a single very long binary vector called a hypervector (here, D = 1024 bits).

The key idea: in very high-dimensional space, two random vectors are almost always "nearly orthogonal" (about 50% of bits differ). This gives the model enormous capacity to hold distinct concepts, and makes it naturally robust to noise and bit errors — flipping a few bits barely changes a vector's identity. Classification becomes simple distance comparison instead of floating-point matrix math.

Why it suits FPGAs / edge hardware:
  • Operations are bitwise (XOR, shift, popcount) — no multipliers.
  • Our design uses 0 DSP blocks, 0 BRAM — pure logic.
  • Massively parallel: 1024 bits processed 64 at a time.
  • Inherent error tolerance → can run at low voltage / aggressive pruning.
Model used here: Binary Spatter Code (BSC). Hypervectors are binary {0,1}; binding is XOR; the similarity metric is Hamming distance (count of differing bits). This is the most hardware-friendly HDC flavour — every primitive maps to cheap gates.

The trade-off: HDC accuracy is usually a little lower than a tuned neural net, but for always-on, ultra-low-power edge inference (wearables, prosthetics, IoT) the energy and simplicity win. That energy/efficiency angle is exactly what this project measures.

4. The application — EMG hand-gesture recognition

The benchmark task is recognising hand gestures from EMG (electromyography) signals — the electrical activity of forearm muscles, measured by 4 electrode channels. This is the same task and dataset as Rahimi et al. (ICRC 2016), the standard reference for HDC-on-EMG, which lets us compare directly with published work. The end use case is gesture-controlled prosthetics, where ultra-low-power always-on inference matters.

How an EMG window becomes a class

  1. A short time window of the 4 EMG channels is captured.
  2. Per channel, signal features are quantised into discrete levels (16 levels).
  3. Each (channel, feature, value) is looked up in item memories (hypervector ROMs).
  4. These are bound + permuted + bundled into one 1024-bit query hypervector.
  5. The query is compared (masked Hamming) to the 5 class prototypes; nearest wins.
"Training" in HDC. There is no gradient descent. For each gesture class we simply bundle all its training windows into one prototype hypervector. Inference is then a single nearest-neighbour search. This one-shot, single-pass training is a major part of HDC's appeal.

5. System architecture — Zyng PS + PL

The Zynq-7020 is a SoC: a hardened ARM Cortex-A9 Processing System (PS) plus FPGA fabric (Programmable Logic, PL) on one chip. Our HDC core lives in the PL; the PS orchestrates data movement. They talk over AXI interfaces.

6. The encoding pipeline & module hierarchy

The encoder implements research-plan Eq. (3.1): a 4 channels × 5 features grid (20 bind operations). Each feature value indexes an item-memory hypervector, which is bound with its channel vector, permuted by feature position, and all 20 are bundled into the query hypervector.

New this month — pruning_mask.sv extracted as a dedicated module. Previously the mask was folded inside popcount_am. It is now a named, parameterised block (matching research-plan §5.3.3) that holds the D-bit mask and gates the Hamming distance, so pruned bit positions never contribute — cutting dynamic energy roughly in proportion to the prune ratio. It supports both a word-addressed AXI write port and a full-width parallel load (so the existing verified board path is untouched). Verified bit-exact: 129/129 checks PASS.

7. Verification — the Python golden reference

The cornerstone of this project's rigour: a bit-exact Python golden model (hdc_ref.py) defines the single source of truth. Every RTL block is checked bit-for-bit against it in automated co-simulation. There is no hand-written "expected" vector — the Python model generates both the stimulus and the expected output, so RTL and reference can never silently drift apart.

9. Research novelty — your actual contribution CONTRIBUTION

Prior FPGA-HDC work mostly prunes the dimension D. The under-explored axis is per-bit position pruning: not how many bits, but which bits. This project's novelty is one main study (Hook A) and two twists, all measured on real Zynq energy.

Twist 1 — informed vs random pruning (the headline result)

At each pruning ratio, compare a discriminability-driven mask (keep bits with the highest Fisher ratio — between-class variance / within-class variance) against a random mask of identical density. The claim: informed pruning beats random by ≥5 percentage points at the same bit-count, proving that bit position matters, not just bit count. This is a clean, defensible scientific result.

Twist 2 — cross-subject mask transferability

Train the mask on one set of EMG subjects, deploy it unchanged on held-out subjects. Either outcome is publishable: if it transfers (within ≤3 pp), there's a universal "bit-importance map" for EMG; if it doesn't, that motivates on-device mask adaptation. (Needs a 36-subject export; currently 5.)

Why this is a good contribution: it isolates an under-studied design axis (which bits to keep), backs it with an interpretable score (Fisher ratio), and measures the payoff in real silicon energy — not just simulation. All claims are relative to the project's own 74.24% baseline, so they stand regardless of the absolute accuracy number.

11. Anticipated questions from the professor (with answers)

On accuracy

Q: Rahimi reports ~90%. Why is your hardware only 74%?
A: We do reproduce ~90% in Python (Stage A 90.36%, Stage B 90.30%). The 74.24% is a different encoder — the hardware-faithful Eq. (3.1) grid — not a degraded version of Rahimi's. The board matches its own golden reference to Δ0.00% over 658k windows, so the system is provably correct. Our contribution is the energy/pruning study on that verified path, and those claims are relative to our own baseline.
Q: Then why not just put the 90% encoder on the FPGA?
A: It's on the roadmap as a possible Stage-B RTL encoder, but it's a richer encoding (more binds / different feature handling) that costs more logic and design time. The current encoder was chosen to lock down a fully verified streaming path first. Crucially, the research questions (which bits to prune, energy vs accuracy) don't depend on the absolute encoder — so we can publish on the 74% path now and upgrade the encoder later.
Q: Earlier you mentioned 59% — what was that?
A: A prototype-training bug — offline class prototypes were built with a saturating 6-bit counter, which clipped when bundling many samples, plus a prototype-load indexing bug. Both are fixed and documented; accuracy went from ~59% to the correct 74.24%. I keep it in the log as evidence of a debugged, honest pipeline.

On the architecture

Q: Why zero DSP and zero BRAM — is that a limitation?
A: It's deliberate and a selling point. HDC is bitwise (XOR / shift / popcount), so it needs no multipliers; item memories are small enough to infer as LUT ROM. It means the design is portable to the cheapest FPGAs and leaves all DSP/BRAM free for other functions — exactly the edge-efficiency story.
Q: You're at 96% slice utilisation at D=1024 — can you even do D=2048?
A: Core-only (out-of-context) D=2048 should fit; the full system with PS+DMA at D=2048 may not, and that boundary is itself a reportable Pareto result. The D-sweep synthesis (scripted, ready to run on the VDI) will give exact LUT/FF/timing per D to answer this with data.
Q: How do you know the RTL is actually correct?
A: Every block is checked bit-for-bit against a Python golden model in automated co-simulation (7 harnesses, up to 1000 cases each). The golden model generates both stimulus and expected output, so there's no hand-written reference to drift. On hardware, the board reproduces the golden exactly over 658k windows.

On the research contribution

Q: What's genuinely novel here vs existing FPGA-HDC papers?
A: Existing work prunes the dimension. We prune bit positions by an interpretable discriminability score (Fisher ratio) and show — on measured silicon energy — that which bits you keep beats random selection at the same count (≥5 pp). Plus cross-subject mask transfer, which asks whether bit importance is universal. Both are new questions for HDC hardware.
Q: Why is informed pruning expected to beat random?
A: Because hypervector bits are not equally informative for class separation. The Fisher ratio ranks each bit by between-class vs within-class variance; keeping high-ranked bits preserves the discriminative signal while dropping noise-like bits. Random pruning removes informative and noisy bits alike. The gap quantifies how much structure HDC encodings actually carry per bit.
Q: What if the cross-subject mask doesn't transfer?
A: That's still a publishable result — it means EMG bit-importance is subject-specific and motivates lightweight on-device mask adaptation. We framed Twist 2 so either outcome is a contribution.
Q: How will you measure energy, and is it credible?
A: An INA219 current sensor across the Vcc_int shunt, logged during batch inference, giving real energy-per-inference — not a vendor estimate. That's the next Phase 3 task and it underpins the whole Pareto.

On scope & timeline

Q: Is this on track for DATE 2027?
A: Bring-up (Phases 1–3) is done and ahead of schedule. Remaining work is the measurement science (D-sweep, energy, Hook A, twists) and the write-up — the August/September blocks in the plan. The hard infrastructure risk is retired.
Q: What's the single biggest risk right now?
A: Two small ones: (1) D=2048 may not close timing with the full PS+DMA system — mitigated because core-only still characterises the axis; (2) cross-subject transfer needs a 36-subject export. Neither blocks the headline Hook A / Twist 1 results.
