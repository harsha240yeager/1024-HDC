Slide 1 — Title

What's on the slide

Project title (1024-bit Hyperdimensional Computing Accelerator on Zynq-7020), the subtitle "streaming, bit-exact-verified HDC classifier for EMG hand-gesture recognition", target venue DATE 2027, and the platform line (ZedBoard, SystemVerilog + Python golden reference).

How to explain it

"This is a hardware accelerator for Hyperdimensional Computing, built and verified on a Zynq FPGA. The two words that define the work are streaming (data flows through at inference rate over DMA) and bit-exact-verified (the RTL matches a Python golden model exactly). The application is EMG hand-gesture recognition, and I'm targeting DATE 2027."

Likely questions

What is the one-sentence contribution?
A measured accuracy/energy/area Pareto for a streaming 1024-bit HDC core on Zynq, plus an informed-vs-random bit-pruning result — not a re-port of prior HDC accuracy.
Why Zyng / ZedBoard specifically?
It pairs an ARM CPU (PS) with FPGA fabric (PL) on one chip, so I can stream data from software to hardware over AXI-DMA and also run a fair ARM-only software baseline on the same silicon.

Slide 2 — The problem & motivation

What's on the slide

Three bullets: edge inference must be cheap/low-power; HDC is a bitwise, brain-inspired alternative to multiply-heavy CNN/MLP; the application is EMG (Rahimi et al. benchmark). A blue box states the thesis: a streaming 1024-bit HDC core with zero DSP/BRAM, where per-bit pruning cuts energy — the contribution being a measured Pareto.

How to explain it

"Always-on edge devices — prosthetics, wearables — can't afford GPU-style multiply-accumulate. HDC replaces that with bitwise operations on long binary vectors, which map beautifully to FPGA logic. I use the EMG gesture dataset from Rahimi's 2016 paper because it's the standard HDC-on-EMG benchmark, so I can compare directly with the literature."

Likely questions

Why not just use a small neural network?
We measured one — a tiny int8 MLP hits ~93% (slide 19). HDC's argument isn't peak accuracy, it's efficiency: bitwise ops, one-shot training, zero DSP, and tolerance to bit errors that lets us prune aggressively. That trade-off is exactly what the Pareto quantifies.
Is HDC actually used in industry, or is it academic?
It's still mostly research, but its appeal for ultra-low-power always-on sensing (biosignals, IoT) is real. The point of this work is to give hardware designers measured numbers on the energy/accuracy trade-off, which is what's missing.

Slide 3 — What is Hyperdimensional Computing?

What's on the slide

Four bullets: every concept becomes one 1024-bit hypervector; random vectors are near-orthogonal in high dimensions (capacity + noise tolerance); classification is nearest-prototype by Hamming distance; the model is Binary Spatter Code (BSC). A green box lists why it suits FPGAs.

How to explain it

"The core idea: in 1024-dimensional binary space, two random vectors differ in about half their bits, so they're 'nearly orthogonal'. That gives huge representational capacity and means flipping a few bits barely changes a vector's identity — inherent error tolerance. Classifying is just finding the nearest stored vector by Hamming distance. I use Binary Spatter Code: binary vectors, XOR binding — the most hardware-friendly flavour."

Likely questions

Why 1024 dimensions — why not more or fewer?
1024 is the sweet spot for this task and it's a clean 16×64-bit word layout for the datapath. But D is a parameter — the D-sweep (slide 14) measures 256 to 2048 so the dimension choice is itself a result, not an assumption.
Where does the noise tolerance actually help you?
It's what makes bit-pruning viable: I can zero out a large fraction of bits in the distance computation and still classify correctly, because the class signal is spread across many bits. That's the basis of Hook A.

Slide 4 — The four HDC primitives

What's on the slide

A table mapping each primitive to its math, meaning, and RTL block: Bind (XOR → xor_permute_top), Permute (cyclic shift → permute_stage), Bundle (majority → bundle_unit), Search (argmin Hamming → popcount_am).

How to explain it

"All of HDC is four operations, and each one is a verified RTL block in my design. Bind (XOR) associates two concepts, like a channel with its value. Permute (a bit rotation) encodes position. Bundle (bitwise majority vote) superposes many vectors into one. Search finds the nearest class prototype. Encoding uses the first three; search does the classification. Every one is XOR, shift, or popcount — no multipliers."

Likely questions

Why is XOR the right binding operation?
XOR is its own inverse and produces a result dissimilar to both inputs, which is exactly the algebra binding needs — you can unbind by XORing again, and bound pairs don't collide with their parts.
Does the majority vote need a tie-break?
Yes — for an even number of bundled vectors, ties are broken deterministically (matching the Python golden), so RTL and reference agree bit-for-bit.

Slide 5 — Application: EMG hand-gesture recognition

What's on the slide

The task (4-channel forearm EMG → gestures, for prosthetics) and the 5-step window-to-class pipeline: capture window → quantise features to 16 levels → look up item memories → bind/permute/bundle → masked Hamming search vs 5 prototypes. A box explains that "training" in HDC is just bundling.

How to explain it

"A short window of the four EMG channels is captured, features are quantised into 16 levels, and each (channel, feature, value) triple indexes a hypervector in an item memory. Those are bound, permuted, and bundled into one query vector, which is compared to the five gesture prototypes. Training is not gradient descent — for each gesture I just bundle its training windows into one prototype. That single-pass training is a big part of HDC's appeal."

Likely questions

How are the item-memory hypervectors generated?
Random seed-42 vectors (committed, reproducible). They're fixed ROMs in hardware — the same seed is used in the Python golden so the two match exactly.
Only 5 gestures / 5 subjects — is that enough?
Five classes is the standard benchmark configuration. Five subjects is the current export; Twist 2 needs a larger 36-subject export, which is flagged as a known dependency (slides 18, 20).

Slide 6 — System architecture: Zynq PS + PL

What's on the slide

The arch.svg block diagram: PS (ARM) streams EMG windows from DDR over AXI-DMA to the PL HDC core, which encodes, classifies, and returns labels. Control is AXI4-Lite; data is AXI4-Stream.

How to explain it

"The Zynq is a system-on-chip: hardened ARM (the PS) plus FPGA fabric (the PL). My HDC core lives in the PL. The ARM sets things up over AXI4-Lite — loads prototypes and the mask, starts the core — then a DMA engine streams windows into the core over AXI4-Stream and results come back. This split is what makes the throughput and the fair software comparison possible."

Likely questions

Why separate control (AXI-Lite) from data (AXI-Stream)?
Control is occasional, register-mapped, latency-insensitive; data is high-rate and throughput-critical. Streaming avoids per-window register writes, which is what gets us to ~216k windows/s.
Is the ARM a bottleneck in the streaming path?
No — once DMA is running, the ARM just manages descriptors. The batch/scatter-gather path (slide 13) keeps the core fed back-to-back.

Slide 7 — The encoding datapath

What's on the slide

The pipeline.svg figure showing Eq. (3.1): a 4-channel × 5-feature grid (20 binds), position-permuted, bundled into the query HV, with the new pruning_mask gating which bits count in the associative-memory distance.

How to explain it

"This is the encoder from the research plan, Equation 3.1. Twenty (channel, feature) pairs are each bound to an item-memory vector, permuted by feature position, and all twenty are bundled by majority into one 1024-bit query. The orange block is the pruning mask — it decides which bit positions are counted in the Hamming distance, and that's the lever for the energy study."

Likely questions

Why 4×5 = 20 binds — where does the 5 come from?
Five features per channel in the Eq. (3.1) encoding. It's the hardware-faithful encoder; it differs from Rahimi's spatial-record encoding, which is the root of the 74-vs-90 accuracy discussion (slide 15).
Does the mask change the encoder output or only the search?
Only the search — the query HV is unchanged; the mask zeroes gated bits before popcount in the AM, so pruning removes switching activity without touching the encode.

Slide 8 — Module hierarchy

What's on the slide

The hierarchy.svg tree of RTL modules. A green box highlights that pruning_mask.sv is newly extracted as a dedicated module (research-plan §5.3.3), holding the D-bit mask and gating Hamming distance, verified bit-exact (129/129 checks PASS).

How to explain it

"This shows how the blocks compose — encoder and associative memory under the core, wrapped by the AXI-Lite and stream interfaces. The thing to note is that I pulled the pruning mask out into its own named, parameterised module this cycle. Previously it was buried inside the popcount unit; now it's a clean block that supports both an AXI word-write port and a full-width parallel load, so the existing verified board path is untouched. It passes 129 of 129 checks."

Likely questions

Why extract the mask into its own module now?
So the pruning study (Hook A) has a clean, independently verifiable interface, and so I can load different masks at runtime without rebuilding the bitstream. It also matches the research-plan module map.
What are the "129 checks"?
The run_pruning_mask_cosim harness — full-mask loads plus AXI word-addressed writes compared bit-for-bit against the Python model.

Slide 9 — Computational microarchitecture (word-serial)

What's on the slide

The compute_datapath.svg: the 1024-bit HV is stored as 16×64-bit words; the AM streams one 64-bit word/cycle — XOR query⊕prototype, AND mask, 64-bit popcount, accumulate distance. A green box explains why word-serial (small popcount + accumulator reuse → 0 DSP, easy timing).

How to explain it

"Rather than a giant 1024-bit comparison in one cycle, I process 64 bits per cycle: XOR the query against a prototype word, AND with the mask word, popcount the result in a LUT adder tree, and accumulate. Sixteen words per class, looped over classes, then argmin. Word-serial means one small popcount and one accumulator are reused — that's how we stay at zero DSP, zero BRAM and close timing at 100 MHz. And because the mask zeroes bits before the popcount, pruning directly removes switching activity — the energy saving is real, not bookkeeping."

Likely questions

Doesn't word-serial hurt throughput vs a parallel tree?
It trades latency for area and timing, but the pipeline hides it across windows — we still hit ~216k windows/s. A full parallel popcount would blow up LUTs and timing for little end-to-end gain.
Why 64-bit words?
Matches the AXI-Stream data width and gives a clean 16-word HV; the popcount tree for 64 bits fits comfortably in LUTs at 100 MHz.

Slide 10 — Cycle-level behaviour: AM FSM & latency

What's on the slide

The compute_fsm.svg and a cycle-count table: encode ~23 cycles, bundle combinational, classify 264 cycles, ≈287 cycles/window ≈ 2.9 µs @ 100 MHz. A box notes CNT_W and prune ratio are the two micro-architectural knobs of Hook A.

How to explain it

"Here's the cycle budget. Encoding a window is about 23 cycles; the masked-Hamming classify over 8 class slots is 264 cycles; roughly 287 cycles per window, about 2.9 microseconds at 100 MHz, and it pipelines across windows. The two knobs that Hook A turns — the bundle counter width CNT_W and the prune ratio — change the datapath's switching and area without changing its structure, so they're clean axes to sweep."

Likely questions

You said 8 classes but only 5 gestures?
The AM is provisioned for 8 class slots (power-of-two indexing); 5 are used for gestures, the rest are unused. The cycle count reflects the provisioned 8.
What does CNT_W actually change physically?
The width of the 1024 saturating counters in the bundler. Narrower counters save area/energy but can saturate and lose accuracy — which is precisely what the sweep measures.

Slide 11 — Verification: the Python golden reference

What's on the slide

The verification.svg flow: a bit-exact Python model (hdc_ref.py) is the single source of truth, generating both the stimulus and the expected output so RTL and reference can't drift.

How to explain it

"This is the methodological backbone. There is no hand-written expected output anywhere. The Python model generates the random stimulus and computes the expected result; the RTL is fed the same stimulus in co-simulation and must match bit-for-bit. So the reference and the hardware physically cannot silently diverge — any drift fails a test immediately."

Likely questions

How do you know the Python model itself is correct?
It's validated against the literature: the same model reproduces Rahimi's ~90% (slide 15, Stage A/B). So the golden is anchored to published results, and the RTL is anchored to the golden.
Is co-simulation as trustworthy as formal verification?
It's directed-random with thousands of cases per block and full end-to-end coverage, then confirmed on real silicon over 658k windows. For this datapath that's strong evidence; formal could be a future add for the control FSMs.

Slide 12 — Verification coverage: 7 + 1 harnesses

What's on the slide

A table of the eight co-sim harnesses and case counts (bind/permute 1000, bundle 500, AM 500, encoder 500, core 500, AXI 200, stream 200, pruning-mask 64), all PASS on Questa and Vivado xsim.

How to explain it

"Every block has its own harness, and there's an end-to-end one and an AXI/stream one, plus the new pruning-mask harness. Case counts run up to a thousand. Everything passes on both simulators — Questa and Vivado's xsim — so the result isn't tool-specific."

Likely questions

Why different case counts per harness?
Larger state spaces get more cases (bind/permute 1000); protocol harnesses (AXI/stream) need fewer because they test sequences, not arithmetic breadth.
Do the harnesses cover back-pressure and stalls?
Yes — run_stream_cosim injects random gaps and back-pressure to prove the stream path is robust, not just the happy path.

Slide 13 — Board results: three measurement paths

What's on the slide

KPI cards (74.24% board accuracy, Δ0.00% vs golden over 658k windows, ~216k win/s, 0/0 DSP/BRAM) and a table comparing Phase 1 (AXI-Lite), Phase 2 (DMA stream), Phase 3 (SG batch) on golden, latency, throughput, timing.

How to explain it

"Three ways to drive the same core, each a data point for the paper. Phase 1 is register-mapped control — the baseline, 3 µs/window. Phase 2 is the DMA streaming path. Phase 3 is scatter-gather batch, the throughput story at ~216k windows/s with positive timing slack. All three pass the 200-case golden, and the full EMG replay matches the golden to zero over 658 thousand windows."

Likely questions

Why is Phase 2 slower per window than Phase 1?
Single-window DMA has descriptor setup overhead; that's why Phase 3 batches many windows through a descriptor ring to amortise it and get the real throughput number.
What does WNS +0.111 ns mean and why care?
Worst negative slack is positive, so the design meets the 100 MHz timing constraint with margin after place-and-route — it's a genuine closed design, not a simulation-only claim.

Slide 14 — Resource utilisation & the D-sweep June gap CLOSED

What's on the slide

Integrated utilisation (35,206 LUT = 66.2%, 96.3% slices, 0 DSP/BRAM) and the OOC core-only D-sweep table (D = 256/512/1024/2048 → LUT, util, WNS, Fmax, cosim PASS). Note: D=2048 = 111% LUT = a reportable Pareto boundary.

How to explain it

"Two things. First, the full system is pure logic — zero DSP, zero BRAM — with the item memory inferred as LUT ROM. Second, this is the D-axis of the Pareto, now measured: I synthesised the core out-of-context at four dimensions and ran functional co-sim at each. LUTs scale roughly linearly with D. D=1024 fits comfortably; D=2048 core-only is 111% of the device — it doesn't fit standalone, and that boundary is itself a result worth reporting."

Likely questions

Why is D=1024's WNS the tightest even though it's not the biggest?
Timing depends on routing/congestion, not just size; D=1024 is the shipping config with the full system, so its critical path is the most stressed. All configs still meet 100 MHz.
Is out-of-context synthesis a fair area measure?
It isolates the core's logic cleanly for the D-axis comparison. For the deployable number I also report the full integrated build (66.2% LUT). Both are on the slide on purpose.

Slide 15 — The two-baseline accuracy story read carefully

What's on the slide

A table of three tracks: Stage A MAP (Python, 90.36%), Stage B BSC (Python, 90.30%), RTL encoder (Python + board, 74.24%). A red "defense one-liner" box.

How to explain it

"This is the slide to be careful and confident on. I report two accuracy numbers deliberately. In Python I reproduce the literature — about 90% — with the same encoding Rahimi used. On the FPGA I run a different, hardware-faithful encoder that lands at 74.24%. These are different algorithms, so they were never supposed to match. The deliverable is a verified deployment path plus an energy/pruning study — not a second copy of the 90% number."
Say the one-liner verbatim: "We reproduced ~90% in Python matching Rahimi; the FPGA runs a different, bit-exact-verified encoder at 74.24%; the contribution is measured energy and informed-pruning Pareto on that verified path — not a second accuracy port."

Likely questions (this slide draws the most)

Rahimi got ~90%. Why is your hardware only 74%?
Because it's a different encoder (Eq. 3.1 grid vs Rahimi's spatial records), not a degraded version. I do reproduce ~90% in Python. The board matches its own golden to Δ0.00% over 658k windows, so the system is provably correct; the contribution is relative (pruning, energy), so the absolute number doesn't affect the claims.
Then why not put the 90% encoder on the FPGA?
It's a richer encoding that costs more logic and design time. I locked down a fully verified streaming path first. The research questions don't depend on the absolute encoder, so I can publish on the 74% path now and upgrade the encoder later — it's on the roadmap.
Isn't reporting 74% risky for a paper?
The opposite — the full 74↔90 gap is traced (encoding, item-memory seed, feature handling), so reviewers see an honest, debugged pipeline. The headline results are relative gaps, which are robust.

Slide 16 — Why 74% is not a weakness (six points)

What's on the slide

Six numbered points: (1) ~90% reproduced in Python; (2) 74% is a different encoder; (3) verification fidelity Δ0.00% over 658k; (4) it's a systems study; (5) headline claims are relative; (6) reporting both is a credibility strength.

How to explain it

"This is the backup for the previous slide — six ways to say the same thing so I can meet whichever angle the question comes from. The anchor is point 3: bit-exact fidelity over 658 thousand real windows. Points 4 and 5 are the framing: this is a hardware/energy study and the claims are relative, so 74 vs 90 is simply the wrong axis to judge it on."

Likely questions

Earlier you mentioned 59% — what was that?
A prototype-training bug: a saturating 6-bit counter clipped when bundling many samples, plus a prototype-load indexing bug. Both fixed and documented; accuracy went to the correct 74.24%. I keep it in the log as evidence of a debugged pipeline.
Which of the six is the strongest point?
Verification fidelity (point 3) — Δ0.00% over 658k windows proves the deployed system is correct, which is the actual engineering deliverable.

Slide 17 — Hook A: 3-axis Pareto contribution

What's on the slide

The novelty statement (prior work prunes dimension; the open axis is which bits) and the axis table: D (DONE, synth+cosim), CNT_W (running, Python sweep), bit-pruning ratio (running, Python sweep).

How to explain it

"This is the main contribution. Existing FPGA-HDC work prunes the dimension D. The under-explored question is not how many bits but which bits. Hook A sweeps three knobs — D, the bundle precision CNT_W, and the bit-pruning ratio — and maps the accuracy/energy/area surface. The D-axis is measured on silicon; the CNT_W and pruning axes are running now as an automated 64-cell Python sweep, with the quick-sanity run already passing. INA219 will add the measured energy axis."

Likely questions

What exactly is the 64-cell grid?
4 dimensions × 4 CNT_W values × 4 keep-ratios = 64 configurations, each evaluated over 5 subjects on the RTL-matched encoder with Fisher-informed masks. Output lands in results/hook_a/.
How long does the sweep take and is it done?
~45–50 h single-core on the VDI; it was ~26% through (up to D=512) at last check, with a checkpointed partial JSON so I can preview the Pareto before it finishes.
How do you get energy without running every cell on hardware?
Accuracy comes from the Python sweep, area from the D-sweep synthesis, and energy from INA219 on a few representative board anchors — I don't need to synthesise all 64.

Slide 18 — Twist 1 & Twist 2

What's on the slide

Twist 1: informed (Fisher-ratio) mask vs random mask at equal density; claim ≥5 pp gap at iso-density. Twist 2: cross-subject mask transfer; either outcome publishable (needs 36-subject export; currently 5).

How to explain it

"Two focused experiments on top of Hook A. Twist 1 is the headline: at the same bit-count, does keeping the most discriminative bits — ranked by Fisher ratio, between-class over within-class variance — beat a random mask? I expect at least a 5-point accuracy gap, which proves bit position carries information. Twist 2 asks whether a mask trained on some subjects transfers to unseen ones. If it does, there's a universal bit-importance map; if it doesn't, that motivates on-device adaptation — either way it's a result."

Likely questions

Why would informed pruning beat random?
Because hypervector bits aren't equally informative. Fisher ranks bits by class separability; keeping high-ranked bits preserves signal while dropping noise-like bits. Random drops both alike. The gap quantifies how much structure the encoding carries per bit.
What if Twist 2 fails to transfer?
That's still publishable — it means EMG bit-importance is subject-specific and argues for lightweight on-device mask adaptation. I framed it so either outcome is a contribution.
Why do you need 36 subjects for Twist 2 but not Twist 1?
Twist 1 is a within-run comparison at one config, fine on 5 subjects. Cross-subject transfer needs enough held-out subjects to be statistically meaningful, hence the larger export.

Slide 19 — Comparison baselines accuracy done

What's on the slide

Table: PL DMA (this work) 74.24% @ ~4 µs; ARM-only HDC 74.15% @ 818 µs (1,222 win/s), 200/200 golden; tiny int8 MLP 93.01% fp / 92.99% int8; AXI-Lite path ~3 µs. Green box: PL DMA ~200× faster than ARM at matched accuracy; INA219 adds energy.

How to explain it

"To make the efficiency argument fair I built the obvious alternatives. The same HDC algorithm in C on the ARM core hits essentially the same accuracy (74.15 vs 74.24) but takes 818 microseconds per window — the FPGA path is about 200× faster. And a tiny int8 MLP reaches 93%, which is the honest 'a small NN is more accurate' point — but it doesn't have HDC's bitwise, zero-DSP, prune-friendly profile. Accuracy for all baselines is done and committed; the energy column comes from INA219 next."

Likely questions

Is the 200× speedup a fair comparison?
Yes — identical algorithm, identical protocol, same silicon (Cortex-A9 vs PL on one Zynq), and the ARM version passes the same 200-case golden. It isolates the hardware acceleration.
The MLP is more accurate — doesn't that undercut HDC?
No — it sets the accuracy reference honestly. HDC's case is efficiency and robustness, which the energy Pareto will quantify. Reporting the MLP strengthens credibility rather than hiding the trade-off.
Why is int8 MLP accuracy basically equal to float here?
Post-training int8 quantisation on a small model with this feature set barely moves accuracy (93.01 → 92.99%); the drop would grow on harder tasks. The number is measured over the full 5-subject TEST split.
